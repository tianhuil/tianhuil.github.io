As the software development life cycle becomes more about prompting and less about coding, organizing and storing prompt data has become increasingly important. At [Global Underdog](https://globalunderdog.com/), we’ve been storing them under the `notes/` folder. However, we have found that the notes folder is getting unruly. While there is a growing concensus that [prompts](https://hamel.dev/blog/posts/evals-faq/how-should-i-version-and-manage-prompts.html) [should](https://launchdarkly.com/blog/prompt-versioning-and-management/) be [versioned](https://genai.stackexchange.com/questions/1725/what-s-the-best-practice-for-storing-prompts-when-using-them-in-code-git-or-da), best practices for doing this have not really been developed. Here is our first

| **Folder**             | **What it does**                                                                                                                                                 | **How does it age?**                                                                                                                                                                 |
| ---------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `notes/reserach`       | Pure research gathering facts from the internet. Helps us audit LLM reasoning.                                                                                   | Facts generally don’t change (e.g. nextjs 16.1 is fixed). No need to update. They are generally only useful for a short period as their insights eventually get baked into the code. |
| `notes/context`        | A series of notes on the external business use cases. They are clear but short to encourage humans to be the primary editor.                                     | Only maintained by humans and never automatically editable by the AI.                                                                                                                |
| `notes/implementation` | These are like asana tasks. They are how we determine what needs to be built and give an AI clear instructions for building them.                                | It is accepted that they go out of date quickly. Their names should have date-time to help us understand. Perhaps they get cleared after a month by a regular process.               |
| `notes/design`         | A series of notes that reflect the current state. Provides high level design consideration and business context. Help both humans and models understand codebase | Should be in sync with codebase. Need to regularly run AI agents that read the codebase and update the docs to keep them accurate.                                                   |

## Happy Path Flow

1. We get new business context so human edits `notes/context` .
2. This requires a new features so an AI does some research and adds it to `notes/reserach`
3. Engineers and PMs discuss implementation, which shows up in `notes/implementation`
4. A PR is generated with new code. Each PR comes with edits to `notes/design` that are kept in sync like code.
